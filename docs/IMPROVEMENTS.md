# 改善点・課題一覧

## 要件の再整理

本プラットフォームの本質は「運送会社同士の助け合い」。荷主も参加可能だが、メインユーザーは運送会社。
2つのモードがある:

1. **ソロモード**: 自社の配車状況を一元管理（車番・車種・運転手・スケジュール）
2. **マルチプラットフォームモード**: 自社配車の一部工程を全体に公開してマッチング

---

## 現状のDB設計 vs 要件のギャップ

### Gap 1: 「会社」の概念がない

**現状**: `users`テーブルの`company`フィールドがただの文字列。
**要件**: 運送会社が主体。会社に複数のドライバーが所属し、会社単位で配車管理する。

**必要な変更**: `companies`テーブルを新設し、`users`が`company_id`で所属する構造にする。会社の住所（本社所在地）、連絡先、信用スコアなどを持つ。

### Gap 2: 「配車計画」の概念がない

**現状**: `trips`テーブルは1便＝1レコード（出発地→到着地）。各便が独立している。
**要件**: 1台のトラックの1日の動きは「往路＋帰路」でセット。例えば:
- トラック③: 横浜→小田原（往路・荷有）、小田原→横浜（帰路・荷無し＝募集中）

これは1つの「配車計画（dispatch_plan）」であり、複数の「区間（leg）」で構成される。

**必要な変更**: `dispatch_plans`テーブル（1日の配車計画）と`trip_legs`テーブル（各区間）の2層構造にする。

### Gap 3: vehicle と trip が紐づいていない

**現状**: `vehicles`テーブルは存在するが、`trips`テーブルとの外部キーがない。`trips.vehicle_type`はただの文字列。
**要件**: トラック①②③のように、具体的な車両（車番・車種）に紐づく配車を管理したい。

**必要な変更**: `dispatch_plans.vehicle_id` → `vehicles.id` のFK。`vehicles`に`plate_number`（車番）をユニーク制約で管理。

### Gap 4: ドライバーと車両の関係が曖昧

**現状**: `vehicles.user_id`でドライバーに紐づくが、1台の車に複数ドライバーが乗る（日替わり）ケースに対応できない。
**要件**: 「運転手名」は配車計画ごとに指定。同じ車両でも日によって運転手が変わる。

**必要な変更**: `dispatch_plans`に`driver_id`（当日の運転手）を持たせる。`vehicles`からは`user_id`を外し、会社所有（`company_id`）にする。

### Gap 5: 「帰り荷募集」の概念がない

**現状**: `trips.trip_type`で`outbound`/`return`を区別しているだけ。「帰り荷が欲しい」という状態を表現できない。
**要件**: 各区間（leg）ごとに「荷有り/荷無し（募集中）/非公開」のステータスが必要。

**必要な変更**: `trip_legs`に`cargo_status`（loaded / empty_seeking / empty_private）を追加。`empty_seeking`の区間だけが公開マッチング対象になる。

### Gap 6: チャット機能がない

**現状**: マッチング後のコミュニケーション手段がゼロ。
**要件**: マッチング成立後にチャットでやり取り。会社情報の漏洩検知も必要。

**必要な変更**: `chat_rooms`と`chat_messages`テーブルを新設。マッチング承認時に自動でチャットルームを作成。

### Gap 7: 地図上の複数ルート同時表示

**現状**: 1便ずつの地図表示のみ。
**要件**: 1つの地図に自社の全トラックのルートを表示し、一覧管理したい。

**必要な変更**: フロントエンドに「配車ボード」画面を新設。dispatch_plans一覧を取得し、全ルートを1つのGoogleMapコンポーネントに重ねて描画。

### Gap 8: 公開範囲のコントロール

**現状**: `trips.is_public`で便単位のON/OFF。
**要件**: 区間（leg）単位で公開/非公開を制御したい。往路は非公開、帰路だけ公開、のような使い方。

**必要な変更**: `trip_legs.visibility`（private / public / company_only）で区間ごとに制御。

### Gap 9: 決済フローの強制がない

**現状**: `approved`状態で「完了」ボタンを押すだけで`completed`にできる。決済をスキップ可能。
**要件**: マッチング成立 → 決済 → 完了、の順序を強制すべき。

**必要な変更**:
- matchステータスに`payment_pending`を追加: `pending → approved → payment_pending → completed`
- バックエンドの`CompleteMatch`ハンドラーで決済済みチェックを追加
- Stripe Webhook成功時のみ`completed`に遷移

### Gap 10: ダッシュボードの表示不備

**現状**:
- 「マッチング済み」カードは`approved`のみカウント。`completed`は表示されない。
- `completed`になったマッチングがダッシュボードのどこにも表示されない。

**必要な変更**: 「完了済み」StatsCardを追加。配送履歴セクションを設ける。

## 課金・マネタイズの課題と解決策

### 課題1: 「直接電話でやりとりされてしまう」問題

運送会社の詳細（会社名・電話番号・住所）を表示すると、プラットフォームを介さず直接取引されてしまう。

**解決策: 段階的情報開示モデル**

| 段階 | 表示する情報 | 条件 |
|------|------------|------|
| 検索結果 | エリア・車種・積載量・料金のみ | 誰でも閲覧可 |
| マッチング申請後 | 会社の評価・実績件数 | ログインユーザー |
| マッチング承認後 | 会社名・チャット開放 | 承認済み＋決済前 |
| 決済完了後 | 電話番号・住所・運転手名 | 決済完了後のみ |

**DB変更**: `companies`テーブルに`display_name`（匿名表示名、例: 「神奈川エリアA社」）を追加。API側で段階に応じたフィールドフィルタリング。

### 課題2: 「荷物がのるか、信用できるか」問題

**解決策: 評価・レビューシステム**

- `reviews`テーブル新設（reviewer_id, reviewee_company_id, match_id, rating, comment）
- マッチング完了後に相互評価を促す
- 会社プロフィールに平均評価・取引件数を表示
- 一定以上の評価で「信頼バッジ」付与

**補助的な対策:**
- 車種・最大積載量はシステムで管理（vehiclesテーブル）
- 荷物重量がvehicle.max_weightを超える場合はマッチング不可にする自動チェック
- 過去のマッチング実績（成功率・キャンセル率）を表示

### 課題3: 「チャットで会社情報を出してしまう」問題

**解決策: コンテンツフィルタリング**

- チャットメッセージ送信時にサーバーサイドでフィルタリング
- `blocked_patterns`テーブル: 電話番号パターン（`\d{2,4}-\d{2,4}-\d{4}`）、メールアドレスパターン、URL、会社名辞書
- 検知時: メッセージを保留にして「連絡先情報が含まれている可能性があります」と警告
- 繰り返し違反: アカウントに警告フラグを立てる

**DB変更**:
- `chat_messages.status`（sent / filtered / warned）
- `user_violations`テーブル（violation_type, message_id, created_at）

## バックエンド改善点

### B1: 権限チェックの不足
**現状**: `CompleteMatch`や`Update`系ハンドラーで、リクエスト元ユーザーが対象リソースのオーナーかチェックしていない。他人の便やマッチングを操作できてしまう。
**改善**: 各ハンドラーで`match.trip.driver_id == currentUser.id`のようなオーナーシップ検証を追加。

### B2: GORMのUpdate問題
**現状**: `repository.Update(id, match)`は構造体の非ゼロ値フィールドをすべて更新する。意図しないフィールドも上書きされるリスク。
**改善**: `db.Model(&Match{}).Where("id = ?", id).Update("status", "completed")`のように、更新対象フィールドを明示する。

### B3: トランザクションがない
**現状**: マッチング承認時にmatchステータス更新とtripステータス更新が別々のクエリ。片方だけ失敗する可能性。
**改善**: `db.Transaction()`でラップし、一貫性を保証。

### B4: エラーハンドリングの統一
**現状**: ハンドラーごとにエラーレスポンスの形式がバラバラ。
**改善**: 共通のエラーレスポンス構造体を定義し、統一的なエラーハンドリングミドルウェアを導入。

### B5: ページネーションがない
**現状**: `GetAll()`で全件取得。データ増加時にパフォーマンス劣化。
**改善**: `limit`, `offset`パラメータを追加。レスポンスに`total_count`, `page`, `per_page`を含める。

### B6: インデックス不足
**現状**: GORMのAutoMigrateに任せており、検索で使うカラムにインデックスが明示的に定義されていない。
**改善**: `trips.driver_id`, `trips.status`, `matches.trip_id`, `matches.shipper_id`, `matches.status`などにインデックスを追加。

## フロントエンド改善点

### F1: MatchCardのフィールド名不一致
**現状**: MatchCardが`match.trip?.origin`と`match.trip?.destination`を参照しているが、APIは`origin_address`と`destination_address`を返す。
**改善**: フィールド名を`origin_address`/`destination_address`に修正。

### F2: MatchCardのmatched_priceフィールド
**現状**: `match.matched_price`を表示しようとしているが、Matchモデルにそのフィールドは存在しない。料金は`match.trip.price`にある。
**改善**: `match.trip?.price`に修正。

### F3: ダッシュボードにcompleted表示がない
**現状**: 「マッチング済み」は`approved`のみカウント。completedは幽霊状態。
**改善**: 「完了済み」StatsCardを追加、配送履歴セクションを追加。

### F4: 状態管理が各ページ独立
**現状**: `useMatches()`を各ページで呼ぶたびに独立したstateが作られる。ページ間でデータが同期しない。
**改善**: React ContextかZustandでグローバルな状態管理に移行。

### F5: レスポンシブ対応
**現状**: モバイル対応の考慮がほぼない。
**改善**: 運送業の現場ではスマホ/タブレットからの利用が主流。モバイルファーストでの再設計が必要。

### F6: リアルタイム更新がない
**現状**: データはページロード時に1回取得するのみ。他ユーザーの操作がリアルタイムに反映されない。
**改善**: WebSocketかSSEで、マッチング通知・チャット・位置情報をリアルタイム配信。

## 優先度マトリクス

| 優先度 | 改善項目 | 理由 |
|--------|---------|------|
| 🔴 最高 | DB再設計（会社・配車計画・区間） | 事業の根幹。現状のDB構造では要件を満たせない |
| 🔴 最高 | 段階的情報開示 | マネタイズの要。これがないとビジネスモデルが成立しない |
| 🟡 高 | チャット機能 | マッチング後のコミュニケーションに必須 |
| 🟡 高 | 決済フロー強制 | 課金をスキップされる穴を塞ぐ |
| 🟡 高 | 権限チェック強化 | セキュリティ上の必須事項 |
| 🟢 中 | 配車ボード画面（複数ルート地図） | ソロモードの核心機能 |
| 🟢 中 | 評価・レビューシステム | 信頼構築に重要だが初期はなくても動く |
| 🔵 低 | ページネーション | データ量が増えてから対応でもOK |
| 🔵 低 | リアルタイム更新 | あると良いがMVPには不要 |
