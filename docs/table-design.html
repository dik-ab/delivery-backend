テーブル設計書

帰り便マッチングプラットフォーム — 新DB設計 v1.0

リレーション概要

  companies ──┬── users (1:N)          会社に複数ユーザーが所属
              ├── vehicles (1:N)       会社が車両を所有
              ├── dispatch_plans (1:N)  会社が配車計画を作成
              ├── subscriptions (1:N)   サブスク契約
              └── reviews (1:N)        会社への評価

  dispatch_plans ──┬── trip_legs (1:N)    配車計画 → 複数区間
                   └── trackings (1:N)   位置情報

  trip_legs ──┬── matches (1:N)          区間に対するマッチング
              └── trackings (1:N)        区間の位置情報

  matches ──┬── payments (1:N)           マッチング → 決済
            ├── chat_rooms (1:1)         承認後にチャット開設
            └── reviews (1:N)            完了後に評価

  chat_rooms ── chat_messages (1:N)      チャットメッセージ
  chat_messages ── user_violations (1:N) 違反記録

companies（会社）

運送会社の基本情報。段階的情報開示の主体。

カラム型制約説明
iduintPK主キー
namevarchar(255)NOT NULL正式会社名（承認後に開示）
display_namevarchar(255)NOT NULL匿名表示名（例: 神奈川エリアA社）
addressvarchar(500)本社住所（決済完了後に開示）
headquarters_latdouble本社緯度
headquarters_lngdouble本社経度
phonevarchar(20)電話番号（決済完了後に開示）
emailvarchar(255)連絡先メール
logo_urlvarchar(500)会社ロゴURL
rating_avgfloatdefault: 0平均評価（マッチング申請後に開示）
total_dealsintdefault: 0取引件数
planvarchar(20)default: 'free'free / standard / premium
created_atdatetime
updated_atdatetime

users（ユーザー）

個人アカウント。運送会社メンバーは company_id で所属。荷主は company_id が NULL の場合あり。

カラム型制約説明
iduintPK
company_iduintFK nullable→ companies.id
emailvarchar(255)UNIQUE NOT NULL
password_hashvarchar(255)NOT NULLbcryptハッシュ
namevarchar(100)NOT NULL
rolevarchar(20)NOT NULLowner / driver / dispatcher / shipper / admin
phonevarchar(20)
is_activebooldefault: true
created_atdatetime
updated_atdatetime

INDEX: idx_users_company (company_id), idx_users_role (role)

vehicles（車両）

会社が所有する車両。配車計画で vehicle_id を指定して使用。

カラム型制約説明
iduintPK
company_iduintFK NOT NULL→ companies.id
plate_numbervarchar(20)UNIQUE車番（横浜 100 あ 1234）
vehicle_typevarchar(20)NOT NULL軽トラ / 2t / 4t / 10t / 大型
max_weightfloat最大積載量 (kg)
statusvarchar(20)default: 'active'active / maintenance / retired
notevarchar(500)
created_atdatetime
updated_atdatetime

dispatch_plans（配車計画）

1台のトラック × 1日 = 1つの配車計画。ソロモードの核心テーブル。

カラム型制約説明
iduintPK
company_iduintFK NOT NULL→ companies.id
vehicle_iduintFK NOT NULL→ vehicles.id
driver_iduintFK NOT NULL→ users.id（当日の運転手）
plan_datedateNOT NULL配車日
statusvarchar(20)default: 'planned'planned / in_progress / completed / cancelled
notetext
created_atdatetime
updated_atdatetime

INDEX: idx_dp_company_date (company_id, plan_date), idx_dp_driver (driver_id), idx_dp_status (status)

trip_legs（区間）

配車計画を構成する各区間。旧 trips テーブルの後継。マッチングの単位。

カラム型制約説明
iduintPK
dispatch_plan_iduintFK NOT NULL→ dispatch_plans.id
leg_orderintNOT NULL区間順序（1, 2, 3...）
origin_addressvarchar(500)NOT NULL出発地住所
origin_latdouble
origin_lngdouble
destination_addressvarchar(500)NOT NULL到着地住所
destination_latdouble
destination_lngdouble
departure_atdatetime出発予定時刻
arrival_atdatetime到着予定時刻
leg_typevarchar(20)default: 'outbound'outbound / return
cargo_statusvarchar(20)default: 'loaded'loaded / empty_seeking / empty_private
cargo_descriptionvarchar(500)積荷内容
cargo_weightfloat現在の積荷重量 (kg)
available_weightfloat空き積載量 (kg)
priceint希望料金（円）
visibilityvarchar(20)default: 'private'private / company_only / public
route_polylinetextエンコード済みポリライン
route_duration_secintdefault: 0ルート所要時間（秒）
route_steps_jsonmediumtextステップごとの情報（位置予測用）
delay_minutesintdefault: 0遅延（分）
statusvarchar(20)default: 'scheduled'scheduled / in_transit / completed / cancelled
created_atdatetime
updated_atdatetime

INDEX: idx_tl_dispatch (dispatch_plan_id), idx_tl_vis_cargo (visibility, cargo_status), idx_tl_departure (departure_at), idx_tl_status (status)

matches（マッチング）

区間に対するマッチングリクエスト。荷主→運送会社、または運送会社→別運送会社。

カラム型制約説明
iduintPK
trip_leg_iduintFK NOT NULL→ trip_legs.id
requester_iduintFK NOT NULL→ users.id（リクエスト送信者）
requester_company_iduintFK nullable→ companies.id（会社間マッチング時）
cargo_weightfloat荷物重量 (kg)
cargo_descriptionvarchar(500)荷物内容
messagetextリクエスト時メッセージ
statusvarchar(20)default: 'pending'下記参照
reject_reasonvarchar(500)拒否理由
request_typevarchar(30)default: 'shipper_to_company'shipper_to_company / company_to_company
created_atdatetime
updated_atdatetime

ステータス遷移

pending → approved → payment_pending → completed
pending → rejected   /   cancelled（どの段階からも）

INDEX: idx_m_trip_leg (trip_leg_id), idx_m_requester (requester_id), idx_m_status (status)
